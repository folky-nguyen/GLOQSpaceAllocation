# 013.01 3D View Recovery And Regression Guards

## Goal

Task `013` already delivers the requested product behavior and the current result is acceptable.

This follow-up exists only to harden the implementation against likely future regressions from the recent review, while changing as little code as possible.

The priority is defensive stability, not new user-facing capability.

## Current Repo State

After `013`, the repo now has:

- `apps/web/src/editor-shell.tsx`
  - local `threeDVisibilityMode`
  - `3D scope` status text
  - current scope persists for the lifetime of the current editor shell session
- `apps/web/src/three-d-viewport.tsx`
  - scope toggle
  - empty-state shortcut to `All Levels`
  - current renderer phase handling for `loading`, `ready`, `empty`, `unsupported`, and `error`
- `apps/web/src/space-scene.ts`
  - derived `vertices` and `edgeVertices`
  - active-floor-only vs all-levels scene filtering
- `crates/render-wasm/src/lib.rs`
  - one fill pass plus one edge pass

The current output is good enough to keep.

The remaining hardening risks are:

- one transient renderer error can leave the 3D view in a sticky error state until remount
- a full document replacement can inherit the previous 3D scope instead of starting from the documented default
- edge readability looks good now, but coplanar line behavior can vary by GPU and browser over time

## Scope

In scope:

- a minimal recovery path for transient 3D renderer errors
- a minimal reset rule for 3D scope when the whole document is replaced
- optional edge-artifact mitigation only if it is reproduced during verification

Out of scope:

- new 3D features
- persistent user preferences
- store refactors
- renderer architecture changes
- generalized retry frameworks
- outline/post-process systems

## Implementation Decisions

### 1. Treat this as hardening only

Do not expand the 3D feature surface.

This task should land only the smallest safeguards needed to keep `013` healthy over time.

### 2. Recover from transient renderer errors without a full page reload

Keep the fix inside `apps/web/src/three-d-viewport.tsx`.

Preferred lean behavior:

- if the viewport enters `error`
- and the scene inputs later change in a meaningful way
- dispose the stale renderer handle
- allow the normal init path to run again

Do not add:

- a generic retry state machine
- background polling
- new global error state

Reason:

- the current failure mode is local to the 3D surface
- recovery should stay local too

### 3. Reset 3D scope only on full document replacement

Keep ordinary browsing behavior unchanged:

- switching between `Plan` and `3D` should preserve the current scope
- switching active levels should preserve the current scope

Reset to `Active Floor Only` only when the whole document is replaced.

In the current repo shape, that primarily means:

- sample-case loads
- any future whole-document load seam that replaces `project` outright

Reason:

- this preserves the documented default for a fresh document context
- it avoids adding more session rules than needed

### 4. Keep the edge-artifact fix gated on reproduction

Do not change the wasm renderer just because the review identified a possible coplanar-line risk.

Only touch `crates/render-wasm/src/lib.rs` if the issue is actually reproduced in manual smoke on a real browser/device.

If reproduction happens, prefer the smallest successful fix:

1. first try one tiny edge-depth or edge-position adjustment
2. keep the same fill + edge pass structure
3. do not add a general outline system

Reason:

- the current visual result is already acceptable
- Rust changes here should be evidence-driven, not speculative

### 5. Keep tests minimal

Do not add UI-heavy tests for this hardening pass by default.

Add tests only if a small pure helper is extracted while implementing:

- recovery input detection
- document-replacement scope reset helper
- edge-offset helper, if one is introduced

Otherwise rely on targeted build plus manual smoke.
