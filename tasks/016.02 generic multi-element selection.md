# 016.02 Generic Multi-Element Selection

## Goal

In `apps/web`, keep the existing pick-many workflow but stop hard-coding the selection shape to spaces only, so the shell can keep selecting spaces today and later absorb additional 3D-selectable element kinds with a smaller follow-up diff.

## Required Outcome

- keep click-based multi-selection working for plan spaces
- keep the current `Select` menu actions working for visible spaces
- replace the space-only selection shape with an element-ref selection shape
- support one selected element and many selected elements through the same generic element seam
- keep the current 2D and 3D space emphasis behavior working with the new seam
- limit current element support to `space`
- leave a clear extension path for future non-space element kinds without introducing them yet

## Constraints

- keep the change local to the web shell
- do not change `ProjectDoc`, persistence, renderer contracts, or Supabase schema
- do not add fake 3D element data just to prove future extensibility
- keep the diff small by reusing current space-specific display logic where it still fits

## Current Repo State

### `apps/web/src/ui-store.ts`

Today `selection` is still modeled as:

- one selected `space`
- many selected `space-set`
- plus view, level, and site-edge branches

That works for spaces today, but adding future selectable element kinds would require duplicating the same pattern for every new kind.

### `apps/web/src/editor-shell.tsx`

Today the shell still assumes space-specific selection branches in multiple places:

- `getSelectionLabel(...)`
- `normalizeSpaceSelection(...)`
- `toggleSpaceSelection(...)`
- selected-space derivation for properties and status
- the selection cleanup effect that repairs invalid or off-level space selections

### `apps/web/src/space-scene.ts` and `apps/web/src/three-d-viewport.tsx`

Today 3D selection emphasis and recovery keys still check:

- `selection.kind === "space"`
- `selection.kind === "space-set"`

These checks should move to a shared element-ref seam so 3D code does not need a new branch every time selection grows another selectable kind.

## Scope

In scope:

- introducing one generic selection-element ref shape
- replacing `space` and `space-set` selection branches with generic selected-element branches
- keeping current space-only helper functions where useful, but deriving them from the generic seam
- updating 3D selection emphasis and recovery-key logic to read from the generic seam
- updating tests that rely on the old selection branch names
- updating `KL.md` if the selection term meaning changes

Out of scope:

- selecting new non-space element kinds
- adding 3D hit-testing or 3D click selection
- changing view, level, or site-edge selection behavior
- redesigning properties or status copy beyond what the new selection seam needs

## Initial Implementation Direction

### 1. Introduce one generic selection-element ref

Lean rule:

- add one discriminated ref type for selectable elements
- start with only one supported element kind: `space`

Reason:

- this keeps current behavior unchanged while creating the extension point the user asked for

### 2. Keep space-specific UI details derived, not authoritative

Lean rule:

- keep helpers such as selected-space lookup and space highlight derivation
- derive them from the generic element selection instead of storing space-specific selection branches directly

Reason:

- the shell still only knows how to display spaces today
- derived space helpers are cheaper than fully generic UI rendering right now

### 3. Move 3D emphasis onto shared helper logic

Lean rule:

- replace direct `space` / `space-set` checks in 3D code with element-ref helper usage

Reason:

- the shared helper becomes the one place future selectable kinds extend from

## Review Update

The first draft was still a little too broad for the size of the current repo. After reviewing the real call sites in `ui-store.ts`, `editor-shell.tsx`, `space-scene.ts`, and `three-d-viewport.tsx`, the leaner implementation path is:

- keep `view`, `level`, and `site-edge` branches unchanged
- replace only the current space-specific authored selection branches with:
  - one selected `element`
  - many selected `element-set`
- define one `SelectionElementRef` type that currently supports only `{ kind: "space"; id: string }`
- keep space-oriented property rows, highlight logic, and menu copy derived from helper functions instead of trying to make the whole UI generic now
- update `KL.md` because the meaning of the recurring `selection` term changes and the old `selectMode` glossary entry is now stale after `016.01`

This keeps the structural future-proofing the user asked for without turning `016.02` into a UI rewrite.
