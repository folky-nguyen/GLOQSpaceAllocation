# 014.01 Site Plan Mixed Case Hardening

This follow-up exists because review of `014` found one real mixed-case data regression and one missing guard that allowed it through.

The fix should stay minimal.

Do not expand the product surface.

Do not add more site-plan features.

Only harden the existing `014` result so the shipped mixed cases stay trustworthy over time.

## Goal

Patch the current `014` mixed-case path with the smallest code change that:

1. restores the broken mixed fixture so all host-level spaces fit inside the default derived building footprint
2. adds one regression guard so future mixed fixtures cannot silently drift outside that footprint

## Review Finding

Current issue:

- `supabase/sample-data/mixed/case-1-single-story-angled-lot.json`
  - `space-lot-a3` includes a point at `{ "xFt": 10, "yFt": 38 }`
  - under the case's default `5 ft` setbacks, that point falls outside the derived building footprint

Why this matters:

- `014` explicitly requires spaces to be drawn inside the building footprint
- the dashboard description in `apps/web/src/test-cases.ts` also promises that behavior
- the current automated tests do not validate sample-case containment, so the regression can recur unnoticed

## Scope

In scope:

- fix the invalid host-level space polygon in mixed case 1
- add one small pure-data regression test for mixed-case containment
- keep the current `Site Plan` UI and renderer behavior unchanged

Out of scope:

- new site-plan editing tools
- generic polygon packing
- auto-repairing user-authored spaces after setback edits
- new runtime validation UI
- API or wasm renderer changes

## Implementation Decisions

### 1. Fix the fixture, not the geometry engine

The failure is in checked-in sample data, not in the current derived-footprint math.

Lean rule:

- patch only the offending polygon points in the mixed fixture
- keep the existing setback derivation algorithm unchanged unless another reproduced bug proves it wrong

Reason:

- the current geometry helpers already pass the intended pure tests
- changing the algorithm would be a much larger and riskier fix than correcting one bad fixture

### 2. Add one sample-case integrity test

Add one small test that loads every mixed fixture and asserts:

- `sitePlan` exists
- a default derived footprint exists
- every point of every space on `sitePlan.levelId` lies inside or on that footprint

Reason:

- this is the smallest durable guard against future mixed-case drift
- it protects the real shipped fixtures, not just synthetic geometry inputs

### 3. Keep the test helper local

Prefer one small helper inside a test file over exporting more runtime helpers just for testing.

Good options:

- extend `apps/web/src/project-doc.test.ts`
- or add one tiny `apps/web/src/test-cases.test.ts`

Pick the smaller diff once implementation starts.

Reason:

- no new runtime seam is needed
- the guard is about sample-data integrity, not product behavior at runtime
