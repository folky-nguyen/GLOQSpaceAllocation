# 009.01 3D View Polish

## Goal

In `apps/web` and `crates/render-wasm`, improve the current 3D view so it looks intentional and professional without changing the core architecture from task `009`.

The upgraded 3D view should feel closer to a lightweight BIM viewer:

- clearer scene depth
- cleaner camera framing
- better material hierarchy
- more polished overlays and empty/loading/error states
- consistent `3D` naming in user-facing text

This task must still stay lean:

- no `three.js`
- no second scene graph
- no new authored 3D model schema
- no general-purpose render framework

## Current Repo State

After task `009`, the repo now has:

- `apps/web/src/space-scene.ts`
  - pure TypeScript scene derivation from `ProjectDoc`
  - prism payloads, scene extents, and default orbit camera
- `apps/web/src/three-d-viewport.tsx`
  - lazy wasm renderer mount
  - orbit/pan/zoom interaction
  - basic loading, unsupported, empty, and error states
- `crates/render-wasm/src/lib.rs`
  - one small WebGPU renderer
  - one solid-color prism path
  - no lighting model beyond baked face tinting

What still feels MVP-only today:

- the scene background is functional but not polished
- materials do not yet read like a deliberate visual system
- the viewport lacks a professional ground/grid treatment
- active and selected emphasis work, but the visual hierarchy is still coarse
- overlay text is useful but still shell-like rather than viewer-like
- naming is mixed between `3d`, `three-d`, and `3D`

## Scope

In scope:

- visual polish for the live 3D scene
- better camera reset and first-frame framing
- stronger active-level and selected-space emphasis
- a professional ground or grid treatment
- cleaner overlay layout and copy
- consistent user-facing `3D` naming
- small renderer refinements that improve readability without changing ownership boundaries

Out of scope:

- walls, doors, windows, slabs, or semantic BIM geometry
- saved camera views
- GPU picking
- shadows, reflections, SSAO, bloom, or post-processing stacks
- animation-heavy transitions that require a permanent render loop
- API or persistence changes

## Implementation Decisions

### 1. Keep the same ownership model from `009`

TypeScript still owns:

- `ProjectDoc`
- scene derivation
- camera fit logic
- overlay/view orchestration

Rust wasm still owns:

- GPU setup
- buffer upload
- draw passes

Do not move authored scene decisions into Rust.

### 2. Prefer visual polish over renderer complexity

This task should improve perceived quality mostly through:

- better palette choices
- clearer depth cues
- better grid/ground reference
- cleaner overlay layout
- better emphasis rules

Do not add:

- physically based materials
- complex light rigs
- a generic material system

### 3. Use `3D` consistently in user-facing copy

Normalize visible labels and task copy to `3D`.

Examples:

- `3D View`
- `3D viewport`
- `3D scene`

For repo paths and CSS/file naming:

- keep lowercase path style when a rename would add churn without real product value
- only rename code/file identifiers if the payoff is clear and the diff stays small

Reason:

- product text should look deliberate
- file paths should stay pragmatic

### 4. Add one professional ground reference

The polished scene should include one low-cost spatial reference:

- a ground plane
- plus either a subtle grid pattern or major/minor guide lines

The ground treatment should:

- sit below the lowest prism
- extend beyond scene extents with margin
- stay visually quiet

Do not introduce a separate environment system.

### 5. Improve material hierarchy with a tiny rule set

Keep one small palette only:

- normal spaces
- active-level spaces
- selected space
- ground/grid

Visual goals:

- normal spaces are calm and slightly desaturated
- active level is clearly emphasized but not neon
- selected space reads immediately as the strongest accent

Do not add per-space authored colors.

### 6. Add edge readability only if it stays cheap

If the scene still reads too flat after palette and ground improvements, add one low-cost edge treatment:

- a line-list edge pass from the same prism data
- or a second color treatment that makes prism boundaries clearer

Do not add a heavyweight outline/post-process pipeline.

### 7. Improve first-frame composition before adding more UI

Refine `getDefaultOrbitCamera(...)` first.

Rules:

- frame from a more architectural angle
- bias target slightly above the vertical center of the model
- keep enough distance that the whole stack reads comfortably
- keep `Fit` deterministic and quick

Reason:

- framing changes quality perception more than extra chrome

### 8. Keep overlays compact and viewer-oriented

The polished 3D view should keep only the information that helps inspection:

- title
- camera hint
- active level
- selection
- space count
- one `Fit` action

Reduce shell-like noise once the renderer is ready.

### 9. No permanent animation loop

Continue to redraw only on:

- scene changes
- camera changes
- resize
- explicit reset/fit

Do not add continuous idle animation for polish.

## UI Behavior

### Ready State

When the 3D view is ready:

- the scene should read clearly at a glance
- depth should be easier to perceive
- the active level should be obvious
- the selected space should stand out immediately

### Empty State

When no spaces can be shown in 3D:

- the viewport should feel intentional, not broken
- the copy should explain why the view is empty
- the layout should still look like a finished product surface

### Unsupported And Error States

When WebGPU is unavailable or renderer init fails:

- the message should be compact
- the copy should be human-readable
- the viewport should still look like a controlled product state

## Testing Plan

Do not add renderer snapshot tests in this task.
