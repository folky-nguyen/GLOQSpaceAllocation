# 007.01 Units Parser Hardening

## Goal

Tighten and extend the units parser in `apps/web` so it accepts the follow-up shorthand forms requested after `007` while staying deterministic and testable.

This follow-up must fix the current shorthand gap and add support for:

- `12'6"` as `12' 6"`
- `12' -6"` as `12' 6"`
- `12 3 3/4` as `12' 3 3/4"`
- `''` as an ASCII alias for `"`

The parser should also anticipate common user entry patterns instead of requiring only one exact punctuation style.

This remains TypeScript-only parser work plus tests and a small inspector update. Do not change persisted document data, Rust wasm, or the API.

## Current Repo State

After `007`, the repo now has:

- `apps/web/src/units.ts`
  - parsing and formatting helpers for imperial lengths
  - a regex-driven parser with three explicit feet-plus-inches patterns
  - current support for:
    - `12'`
    - `12' 6"`
    - `12'-6 1/2"`
    - `7 1/4"`
    - `9.5'`
- `apps/web/src/units.test.ts`
  - pure tests for parsing, formatting, conversions, and round-trip behavior
- `apps/web/src/units-inspector.tsx`
  - a manual diagnostic panel driven by `units.ts`

What is still missing or too strict today:

- `12'6"` currently fails
- `12' -6"` currently fails
- markerless `12 3 3/4` currently fails
- `''` is not normalized to inches
- the parser does not yet define a disciplined fallback strategy for partially marked or shorthand user input

## Updated Context

- The product UI is still US imperial ft-in only.
- The editor already has a `Unit` inspector, so parser behavior can be verified both through unit tests and quick manual smoke checks.
- The current parser is strict enough to reject common CAD/Revit-style shorthand, which is a usability bug for future dimension editing.
- This follow-up is about input tolerance, not about changing canonical storage or display formatting.

## Scope

In scope:

- hardening `parseFeetAndInches(...)`
- adding one normalization pass before parsing
- adding explicit shorthand fallback rules for markerless multi-token input
- treating `''` as inches
- expanding parser tests
- updating the inspector sample chips/help text to include the new shorthand examples

Out of scope:

- changing `formatFeetAndInches(...)` output style
- adding metric parsing
- adding locale-aware decimal separators
- accepting arbitrary natural language like `12 feet 6.5 inches`
- changing project document fields away from decimal feet

## Implementation Decisions

### 1. Keep the parser deterministic by using a strict parse order

Parse in this order:

1. normalize the raw text
2. extract one optional leading sign
3. attempt explicit unit-marked parsing
4. attempt shorthand markerless parsing
5. return `null` on failure

Reason:

- explicit input should always win over heuristics
- shorthand should be a fallback, not the primary grammar
- this avoids accidental reinterpretation of already-valid explicit strings

### 2. Add one normalization phase before any grammar matching

Add a small internal normalization helper in `apps/web/src/units.ts`.

Normalization steps:

- trim outer whitespace
- normalize Unicode prime-style and smart-quote style unit markers to ASCII
  - prime-like and apostrophe-like characters -> `'`
  - double-prime-like and double-quote-like characters -> `"`
- replace each non-overlapping `''` pair with `"`
- collapse repeated internal whitespace to a single space

Examples after normalization:

- a Unicode-prime feet/inches string -> `12' 6"`
- `7''` -> `7"`
- `12  3   3/4` -> `12 3 3/4`

Do not silently normalize arbitrary punctuation beyond these explicit cases.

### 3. Relax explicit feet-plus-inches separator handling

Explicit feet-plus-inches parsing should accept all of these equivalent forms:

- `12'6"`
- `12' 6"`
- `12'-6"`
- `12'- 6"`
- `12' -6"`
- `12' - 6"`
- `12'6 1/2"`
- `12' 6 1/2"`
- `12'-6 1/2"`
- `12' - 6 1/2"`

Separator rule:

- after the feet mark `'`, accept zero or more spaces
- optionally accept one hyphen
- then accept zero or more spaces
- then parse the inches component

This is the concrete fix for the review finding in `007`.

### 4. Add a markerless shorthand grammar for common typed values

If explicit parsing fails and the normalized input has no `'` or `"` markers left, parse these markerless forms:

- two integer tokens:
  - `12 6` -> `12' 6"`
- three tokens with trailing fraction:
  - `12 3 3/4` -> `12' 3 3/4"`
- two tokens where the second token is a fraction:
  - `7 1/4` -> `7 1/4"`

Token interpretation rules:

- `a b`
  - if `b` is an integer, treat as `feet inches`
- `a b/c`
  - treat as `inches fraction`
- `a b c/d`
  - treat as `feet inches fraction`

This keeps the shorthand compact but still deterministic.

### 5. Keep markerless shorthand deliberately narrower than explicit syntax

To avoid ambiguous or surprising interpretations, markerless shorthand should reject:

- single bare numbers like `12`
- single bare decimals like `9.5`
- four or more numeric tokens
- markerless decimals in the feet token
  - reject `12.5 6`
- markerless decimals in the inches token
  - reject `12 3.5`

Reason:

- single-number input is ambiguous between feet and inches
- decimal shorthand without markers quickly becomes guesswork
- the dedicated feet-format section in the inspector already covers decimal-feet entry cleanly

### 6. Continue to allow inch overflow and normalize by total inches

Whether the input is explicit or shorthand, inch values above `11` should still parse and normalize through total-inch arithmetic.

Examples:

- `14"` -> `14 / 12` feet
- `5'14"` -> `6' 2"`
- `5 14` -> `6' 2"`
- `12 15 3/4` -> `13' 3 3/4"`

This keeps the parser forgiving without changing internal storage.

### 7. Keep the sign rule simple

Continue to allow exactly one optional leading `-` sign for the whole value.

Accepted:

- `-12'6"`
- `-12 3 3/4`
- `-7''`

Rejected:

- `12' -6"` as a negative inch token
- `12 - 3 3/4` as a markerless negative middle token
- `--12'`

The sign still applies to the whole length only.

### 8. Treat `''` as an input alias only, not a formatting output

The formatter should continue to emit the canonical inch mark `"` and never emit `''`.

Reason:

- `''` is a user-entry convenience
- canonical output should remain normalized and unambiguous

### 9. Update the inspector to make the new accepted forms discoverable

Update `apps/web/src/units-inspector.tsx` so the manual diagnostic surface reflects the hardened parser.

Inspector changes:

- add shorthand sample chips for:
  - `12'6"`
  - `12 3 3/4`
  - `7''`
- add one compact helper note indicating:
  - `''` is accepted as inches
  - markerless `ft in frac` shorthand is accepted

Do not turn the inspector into a documentation-heavy panel. One short note is enough.

## Input Strategy

This follow-up should explicitly anticipate realistic user entry patterns.

### Accepted user-friendly forms

- explicit feet only:
  - `12'`
  - `9.5'`
- explicit feet + inches:
  - `12'6"`
  - `12' 6"`
  - `12'-6"`
  - `12' -6"`
  - `12'-6 1/2"`
- explicit inches only:
  - `7"`
  - `7 1/4"`
  - `7''`
- markerless shorthand:
  - `12 6`
  - `12 3 3/4`
  - `7 1/4`
- normalized copy/paste glyphs:
  - a Unicode-prime feet/inches string

### Rejected ambiguous forms

- `12`
- `9.5`
- `12 3.5`
- `12.5 6`
- `12'6`
- `1 2 3 4`

These should stay invalid because the parser should be forgiving, not guessy.
