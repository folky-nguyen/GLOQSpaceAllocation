# 017.01 Shared Level Headers And Aligned Actions

## Goal

In `apps/web`, redesign the `Levels` portion of the existing `Level Manager` so the desktop layout reads like one compact grid:

- show one shared header row for `Name`, `Elevation`, and `Height`
- stop repeating those labels inside every level row
- keep the left activate button column and right action buttons vertically aligned across rows

This is a follow-up to `017` and should stay focused on the level-list presentation, not on broader panel behavior.

## Required Outcome

- the `Levels` section shows one desktop header row for `Name`, `Elevation`, and `Height`
- each level row aligns to the same column tracks as that header
- the `Activate` / `Active` button column is the same width on every row
- the `Up`, `Down`, and `Delete` buttons line up cleanly across rows
- current CRUD, reorder, active-level, and feet-inch behaviors stay unchanged

## Constraints

- keep the change local to `apps/web/src/editor-shell.tsx` and `apps/web/src/styles.css`
- prefer reshaping the existing markup over extracting a new component
- do not change `ProjectDoc`, level mutation helpers, renderer contracts, API behavior, or Supabase schema
- keep the diff small and avoid a full visual redesign of the whole panel

## Current Repo State

### `apps/web/src/editor-shell.tsx`

Today the `Levels` section renders:

- one `level-row-activate` button per row
- one `level-row-main` grid per row
- repeated field labels inside every row for:
  - `Name`
  - `Elevation`
  - `Height`
- one trailing `level-row-actions` group per row

The `Name` and `Elevation` inputs already have explicit `aria-label` values, so the visible repeated labels are mainly a layout concern rather than the only accessibility surface.

### `apps/web/src/styles.css`

Today the row layout is a row-level grid nested around a second internal grid:

- `.level-row`
- `.level-row-main`
- `.level-row-actions`

That structure works for the current compact card layout, but it still reads as repeated small forms rather than one scan-friendly table-like list.

## Initial Review Update

Reviewed against the supplied markup and the annotated screenshot.

The next improvement is not general spacing any more. It is row consistency:

- repeated `Name`, `Elevation`, and `Height` labels create visual noise
- row content no longer lines up to one shared reading line across the list
- the request is specifically asking for a single header treatment plus straighter button columns

Because of that, `017.01` should change the list structure a little more than `017`, but still stay inside the existing `LevelManager` owner seam.

## Scope

In scope:

- add one shared desktop header row above the levels list
- remove repeated desktop labels from each row
- align the activate column and trailing actions by fixed column sizing
- keep the current active-row highlight and existing controls

Out of scope:

- redesigning `Auto-generate` or `Defaults`
- changing button text unless alignment requires a very small copy tweak
- changing level behavior, validation, or document state ownership
- moving `LevelManager` into another file

## Implementation Decisions

### 1. Treat the desktop list like one grid, not a stack of mini forms

Lean rule:

- add one shared header row above `.level-manager-list`
- make the header and each row use the same desktop column tracks:
  - activate
  - name
  - elevation
  - height
  - actions

Reason:

- this is the simplest way to match the requested screenshot behavior
- shared column tracks create the button alignment the current nested layout cannot guarantee cleanly

### 2. Remove only the repeated visible labels, not the field meaning

Lean rule:

- remove the repeated desktop `Name`, `Elevation`, and `Height` captions from each row
- keep explicit input `aria-label` values so the inputs still have stable names

Reason:

- the user wants the repetition gone from the visible layout
- accessibility should not regress just because the labels move to a shared header

### 3. Use fixed action tracks so the buttons stay vertically aligned

Lean rule:

- keep `Up`, `Down`, and `Delete` in one dedicated trailing column group
- size that group with stable button widths instead of content-driven drift

Reason:

- the screenshot feedback is specifically calling out button alignment
- fixed tracks are a smaller and clearer solution than ad hoc margins per row

### 4. Preserve a readable narrow-screen fallback

Lean rule:

- keep or restore a stacked mobile layout when the current responsive breakpoint collapses the list
- if the shared header is desktop-only, hide it on narrow screens where stacked fields read better

Reason:

- one global header helps desktop scanning
- narrow screens still need row-level readability after the desktop labels are removed

## Acceptance

- on desktop, the `Levels` section shows one shared header row for `Name`, `Elevation`, and `Height`
- desktop rows no longer repeat those visible labels
- the left activate buttons and right action buttons align across the full list
- current editing and reorder behavior still works
- narrow-screen readability is preserved
