# 007.02 Decimal-Inch And Default-Unit Parsing

## Goal

Extend the `apps/web` imperial length input layer so it accepts the next two user-entry patterns while keeping internal storage in decimal feet:

- explicit decimal inches such as `1.2"` and `1.2''`
- single bare numeric input such as `1.24`, interpreted by the assigned parameter unit

Bare single-number input should default to feet so `1.24` means `1.24 ft` unless the caller explicitly says the assigned unit is inches, in which case `1.24` means `1.24 in`.

This remains TypeScript-only UI-boundary parsing work plus tests and a small inspector update. Do not change persisted document data, Rust wasm, or the API.

## Current Repo State

After `007.01`, the repo now has:

- `apps/web/src/units.ts`
  - `parseFeetAndInches(input: string): number | null`
  - explicit feet parsing
  - inches-only parsing
  - markerless shorthand parsing like `12 6` and `12 3 3/4`
  - inches aliases like `''`
- `apps/web/src/units.test.ts`
  - positive coverage for explicit imperial forms, shorthand, normalization, overflow, and round-trips
  - rejection coverage that still treats bare single-number input like `12` and `9.5` as invalid
- `apps/web/src/units-inspector.tsx`
  - a manual parser diagnostic panel driven directly by `parseFeetAndInches(...)`
- `apps/web/src/editor-shell.tsx`
  - current live length-entry commit handlers for level elevations and default story height
  - all current real inputs are still feet-oriented

What is still missing today:

- `1.2"` fails because the inch component parser only accepts:
  - whole inches
  - simple fractions
  - whole inches plus simple fractions
- `1.2''` fails for the same reason after `''` normalization turns it into `1.2"`
- bare single-number input like `1.24` still fails because the parser intentionally rejects one-token markerless values
- the parser has no caller-supplied unit hint for fields whose assigned UI unit is inches instead of feet

## Updated Context

- TypeScript still owns all unit parsing and formatting at the UI boundary.
- Internal editor values must remain plain decimal feet.
- The current repo does not yet expose a generic parameter-metadata layer that carries an assigned display unit into every input field.
- The smallest useful seam for this follow-up is therefore a parser option, not a new document field or a new UI-state system.
- Explicit unit markers must continue to win over heuristics:
  - `1.2"` is inches even if the caller default is feet
  - `1.24'` is feet even if the caller default is inches
- The request is input tolerance only. It does not require changing formatter output into decimal inches; normalized display can remain canonical imperial output.

## Scope

In scope:

- extending explicit inch parsing to accept decimal inches
- allowing a single bare numeric token to parse using a caller-supplied default unit
- making feet the default bare-number unit when no option is supplied
- keeping explicit `'`, `"`, and `''` markers authoritative over the default unit
- expanding pure parser tests
- updating the units inspector so the new input modes can be manually exercised

Out of scope:

- changing canonical storage away from decimal feet
- changing `formatFeetAndInches(...)` output style to decimal inches
- adding metric parsing
- introducing a new parameter-schema or metadata system
- changing Rust wasm, API, or Supabase persistence

## Implementation Decisions

### 1. Extend the existing parser surface with one small unit hint

Keep the current public parser seam and add one optional argument:

```ts
export type DefaultLengthUnit = "ft" | "in";

export function parseFeetAndInches(
  input: string,
  options?: { defaultUnit?: DefaultLengthUnit }
): number | null;
```

Behavior:

- if `options.defaultUnit` is omitted, treat bare single-number input as feet
- if `options.defaultUnit` is `"in"`, treat bare single-number input as inches
- explicit unit-marked input still ignores the default and uses the marker

Reason:

- this is the smallest change that supports assigned-unit parsing without adding a second parser or a parameter-metadata abstraction
- current feet-oriented call sites continue to work with no or minimal churn

### 2. Allow decimal inches only where the grammar is already explicitly in inches

Update the inches-component parser so it also accepts one unsigned decimal token.

Accepted examples:

- `1.2"`
- `1.2''`
- `12' 1.2"`
- `12'-1.2"`

Still reject:

- `1.2 1/4"`
- `1.2 3/4"`
- markerless multi-token decimal shorthand such as `12 1.2`

Reason:

- the explicit inch marker removes ambiguity
- mixed decimal-plus-fraction inch syntax adds complexity without a user request
- markerless decimal shorthand remains guessy and should stay out of scope

### 3. Add a single bare-number fallback after explicit and shorthand parsing

Parsing order should become:

1. normalize the raw input
2. extract one optional leading sign
3. try explicit unit-marked parsing
4. try existing markerless shorthand parsing
5. if the input is exactly one unsigned numeric token, apply the caller default unit
6. otherwise return `null`

Bare-number rule:

- accept one integer or decimal token
- examples:
  - `1.24` with default feet -> `1.24 ft`
  - `1.24` with default inches -> `1.24 in`
  - `12` with default feet -> `12 ft`
  - `12` with default inches -> `12 in`

Reason:

- the user request explicitly changes the old `007.01` rule that rejected single bare numbers
- keeping the fallback limited to one numeric token avoids destabilizing existing shorthand heuristics

### 4. Keep the parser deterministic and precedence-driven

Precedence must remain:

- explicit markers first
- shorthand second
- bare-number default-unit fallback last

Examples:

- `1.2"` with default feet -> inches
- `1.24'` with default inches -> feet
- `7 1/4` -> existing shorthand inches behavior
- `12 6` -> existing shorthand feet-plus-inches behavior
- `1.24` -> default-unit fallback only because no earlier grammar matched

This keeps the new behavior additive instead of rewriting older accepted forms.

### 5. Do not broaden formatting in this follow-up

`formatFeetAndInches(...)` should stay canonical and fraction-based.

That means:

- decimal-inch input is accepted on entry
- normalized display can still round to the configured inch denominator
- `1.2"` may normalize to the nearest canonical fractional-inch display string instead of echoing `1.2"`

Reason:

- the user asked for input acceptance, not a new decimal-inch display mode
- keeping output canonical avoids expanding the formatting surface without a stronger product need

### 6. Use the inspector to validate both default-unit modes

Update `apps/web/src/units-inspector.tsx` so manual validation covers:

- explicit decimal-inch samples:
  - `1.2"`
  - `1.2''`
- one bare-number sample:
  - `1.24`
- a small selector or toggle for bare-number default unit:
  - feet
  - inches

Behavior:

- switching the default unit changes only the interpretation of bare single-number input
- switching the default unit does not change explicit unit-marked results

This gives one concrete place to verify the new contract before inch-assigned product fields exist.

### 7. Keep current product inputs on default feet

The live editor inputs in `apps/web/src/editor-shell.tsx` are still feet-oriented.

For this follow-up:

- existing story-height and level-elevation inputs should continue to parse as feet by default
- no new inch-assigned field needs to be added just to prove the parser contract

Future inch-assigned fields can pass `defaultUnit: "in"` when they appear.

## Input Strategy

This follow-up should support these entry patterns:

### Accepted

- explicit feet:
  - `1.24'`
- explicit inches:
  - `1.2"`
  - `1.2''`
- explicit feet plus decimal inches:
  - `12' 1.2"`
  - `12'-1.2"`
- existing shorthand:
  - `12 6`
  - `12 3 3/4`
  - `7 1/4`
- bare single numeric token:
  - `1.24` with default feet
  - `1.24` with default inches

### Rejected

- mixed decimal-plus-fraction inches:
  - `1.2 1/4"`
- markerless decimal multi-token shorthand:
  - `12 1.2`
- malformed markers:
  - `12'6`
  - `12'''`
