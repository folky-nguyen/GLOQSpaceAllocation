# 012.01 Draggable Test Dashboard And Polygon Apartment Cases

This note supersedes `012 test menu and sample case fixtures.md` for the current implementation direction.

## Goal

Replace the flyout-style `Test` menu plan with one movable in-app test dashboard.

When the user clicks `Test`, the editor should open one floating dashboard window that can be dragged around the workspace.

Inside that dashboard:

- one slicer-style dropdown for `Level`
- one slicer-style dropdown for `Space`
- one explicit `Mixed Cases` section with three fixed presets

Clicking a dropdown option or a mixed preset should immediately generate or load that validation case into the local editor state.

The mixed presets must include:

1. `Case 1`: only `1` above-grade story
2. `Case 2`: `1` basement and `3` above-grade stories
3. `Case 3`: `3` basements and `12` above-grade stories

In every mixed case, apartment spaces must be packed tightly in different adjacency patterns, and the space footprints must be polygons rather than simple squares or rectangles.

All sample case data stays under `supabase/sample-data/` and remains future-compatible with Supabase snapshot persistence.

## Implementation Decisions

### 1. Use a draggable dashboard, not a flyout menu

The `Test` button opens one overlay dashboard inside the workspace.

Rules:

- draggable by header
- one instance only
- not resizable in the first version
- stays open after a case loads so the user can switch cases quickly

### 2. Keep two slicers plus three fixed mixed presets

The dashboard should have three validation surfaces:

- `Level`
  - loads a level-focused case
- `Space`
  - loads a space-focused case
- `Mixed Cases`
  - loads one of the three required combined building scenarios

Do not hide the mixed presets behind another nested menu.

### 3. Keep sample data as whole snapshot-compatible JSON documents

Do not split runtime case content into a second schema.

Each JSON case should remain one whole `ProjectDoc`-compatible document so it can map cleanly to future persisted snapshots.

Use a small TypeScript manifest only for:

- labels
- grouping
- preferred active level
- preferred view
- descriptive notes

### 4. Make polygon footprints canonical in `ProjectDoc`

Replace rectangular-only authored space geometry with polygon footprints.

Target shape:

```ts
type Point2Ft = { xFt: number; yFt: number };

type Space = {
  id: string;
  levelId: string;
  name: string;
  footprint: Point2Ft[];
};
```

Rules:

- all points stay in decimal feet
- rectangles are just 4-point polygons
- area, bounds, label anchor, and render extents are derived from the polygon

Reason:

- the user explicitly wants non-rectangular apartment spaces
- TypeScript must remain the owner of geometry truth

### 5. Keep polygon math in TypeScript, not Rust

TypeScript should own:

- polygon area
- polygon bounds
- polygon centroids or label anchors
- triangulation or derived mesh generation needed for rendering

Rust wasm should continue to own GPU rendering only.

Reason:

- this preserves the repo architecture contract
- it avoids a second geometry authority in Rust

### 6. Render polygons in 2D through one SVG plan layer

Do not fake polygon spaces with clipped HTML buttons.

Use one SVG-based plan layer so each space can render as:

- one polygon path
- one selection state
- one label anchored from derived geometry

### 7. Render polygons in 3D as extruded prisms

The 3D payload should move from box extents to polygon-based prism meshes.

Each visible space should derive:

- floor polygon vertices
- extruded side faces
- top faces
- emphasis or selection state

### 8. Lock the initial case coverage now

The first implementation must include:

- `3` level cases
- `3` space cases
- `3` mixed cases

Required level cases:

- `Single Story`
- `1 Basement + 3 Stories`
- `3 Basements + 12 Stories`

Required space cases:

- one single-floor apartment layout with staggered polygon units
- one single-floor apartment layout with offset polygon units around a shared core
- one single-floor apartment layout with denser polygon packing and varied edge angles

Required mixed cases:

- `Case 1`: `1` story with one tight polygon apartment arrangement
- `Case 2`: `1` basement + `3` stories with a second, clearly different arrangement
- `Case 3`: `3` basements + `12` stories with a third, clearly different arrangement

## Dashboard Behavior

### Open

- click `Test`
- dashboard opens inside the workspace
- initial position should avoid colliding with the level manager by default

### Move

- drag by header only
- keep the window inside the visible workspace bounds

### Load

When a case is chosen:

1. load the JSON case
2. replace the current local `ProjectDoc`
3. repair `activeLevelId`
4. reset selection to the preferred case focus
5. switch to the preferred view when the case metadata asks for it
6. keep the dashboard open
