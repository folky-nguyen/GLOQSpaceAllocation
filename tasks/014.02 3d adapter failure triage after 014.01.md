# 014.02 3D Adapter Failure Triage After 014.01

This follow-up exists because, after `014.01`, the user reported another failure while opening `3D View`.

The original screenshot showed:

- case `mixed-case-3`
- `3D View`
- renderer state banner: `3D renderer error`
- error text beginning with `Failed to request WebGPU adapter: No suitable graphics adapter found ...`

New evidence now shows:

- all `3` mixed cases fail in `3D View`
- both `Active Floor Only` and `All Levels` can hit the failure path
- one screenshot shows a prolonged `Starting 3D renderer` state before any successful render
- another screenshot shows the later adapter error state for a different case and scope

The fix should stay lean.

Do not assume `014.01` is the root cause just because the issue was seen afterward.

Check first, then either:

- fix the real cause with the smallest safe diff
- or revert only the specific change that is proven to have caused the regression

## Goal

Determine whether the current failure is:

1. a case-independent `3D View` startup failure unrelated to the `014.01` fixture patch
2. a WebGPU adapter-availability problem on the current browser or machine
3. a renderer-artifact or rebuild issue caused by the current `crates/render-wasm/pkg` startup path
4. a browser-side classification issue where a no-adapter condition is being shown as a generic `error` instead of a stable `unsupported` state

Then land the smallest acceptable resolution.

## Current Evidence

### 1. The failure is now confirmed across all mixed cases, so it is not sample-case-specific

`014.01` only changed:

- `supabase/sample-data/mixed/case-1-single-story-angled-lot.json`
- `apps/web/src/project-doc.test.ts`
- `tasks/014.01 site plan mixed case hardening.md`

It did not change:

- `apps/web/src/three-d-viewport.tsx`
- `apps/web/src/editor-shell.tsx`
- `crates/render-wasm/src/lib.rs`

The new user report says all `3` mixed cases fail in `3D View`.

That makes a direct regression from the `014.01` case-1 polygon-point patch even less likely.

### 2. The screenshots point at the async renderer-init path, not at scene generation

One screenshot shows:

- `3D View`
- visible `Spaces` count already populated
- state banner still at `Starting 3D renderer`

That means:

- `buildSpaceScenePayload(...)` already produced visible scene items
- the failure happens after scene derivation
- the likely fault surface is wasm renderer initialization, adapter request, or a never-resolving startup path

### 3. Current 3D code treats only missing `navigator.gpu` as `unsupported`

Today, `apps/web/src/three-d-viewport.tsx` only sets phase `unsupported` when the browser does not expose `navigator.gpu`.

Any wasm-side renderer init failure after that falls into the generic `error` phase.

That means a browser/device that exposes WebGPU but cannot provide a usable adapter currently shows a harder error path than it likely should.

### 4. The wasm layer throws the reported adapter failure before the first successful render

`crates/render-wasm/src/lib.rs` requests a WebGPU adapter during `create_renderer(...)`.

The screenshot message matches that adapter request failure.

That happens before any successful render, so the case-1 containment patch from `014.01` is not the most likely cause.

### 5. The build helper always regenerates `crates/render-wasm/pkg` when `wasm-pack` is available

`setup/ensure-render-wasm.mjs` rebuilds `crates/render-wasm/pkg` whenever `wasm-pack` exists on the machine.

That means a post-`014.01` runtime change can still come from:

- the current generated wasm package
- the current renderer startup behavior

even though `014.01` itself did not edit renderer source files.

## Scope

In scope:

- verify whether the reported failure is case-independent and unrelated to the `014.01` fixture patch
- classify the current failure as adapter issue vs renderer-artifact issue vs UI classification issue
- fix the smallest real cause
- revert only if rollback is proven to be the safest minimal resolution
- keep the user-facing 3D state stable when no adapter is available

Out of scope:

- new 3D features
- WebGL fallback
- renderer architecture changes
- speculative Rust renderer rewrites
- reverting `014.01` without evidence

## Implementation Decisions

### 1. Evidence-first triage

Do not revert `014.01` first.

Reason:

- `014.01` fixed a real sample-data bug
- blind rollback would reintroduce known-bad fixture data
- all three cases now fail, which does not match the code surface changed by `014.01`

### 2. Treat no-adapter failures as `unsupported` if that is what reproduction confirms

If the browser exposes `navigator.gpu` but `create_renderer(...)` fails with:

- `No suitable graphics adapter found`
- `webgpu found no adapters`
- equivalent no-adapter wording

then the preferred minimal fix is:

- keep the existing wasm init path
- classify that specific message as `unsupported` in `apps/web/src/three-d-viewport.tsx`
- show the stable `WebGPU unavailable` state instead of a generic renderer crash state

Reason:

- this is smaller than a renderer rewrite
- it matches the product meaning of the failure more accurately
- it avoids blaming the sample-case flow for an adapter-availability problem

### 3. Check the generated wasm package path before blaming the sample data

If the reproduction is case-independent, inspect whether the current `crates/render-wasm/pkg` startup path is the real regression surface.

Lean rule:

- prefer checking current renderer startup behavior before touching sample fixtures
- if needed, restore the last known-good renderer artifact or startup behavior
- do not roll back the `014.01` fixture correction unless evidence points there directly

Reason:

- `014.01` changed data for one case only
- the current symptom affects all cases
- `web build` and some dev flows can regenerate `crates/render-wasm/pkg`

### 4. Keep the `014.01` fix unless a rollback is proven necessary

Rollback is acceptable only if reproduction shows that:

- reverting the `014.01` runtime-affecting change removes the failure
- and the failure is truly caused by that change rather than by adapter availability

At the moment, that is not the leading hypothesis.

### 5. Touch Rust only if TypeScript-side classification or artifact restore cannot solve the issue

Only edit `crates/render-wasm/src/lib.rs` if:

- the adapter failure needs a more reliable machine-readable message
- or a browser-side classification fix cannot distinguish unsupported from unexpected errors safely
- or the current renderer startup behavior is proven broken at the source level

Otherwise keep Rust unchanged.

### 6. Touch `setup/ensure-render-wasm.mjs` only if forced rebuild is the proven trigger

If evidence shows:

- the checked-in pkg works
- but the auto-regenerated pkg fails on this machine or in this workflow

then the minimal acceptable fix may be to stop forcing a rebuild in the affected path and reuse the known-good checked-in pkg.

Do not make this change without reproduction evidence.

## Lean Investigation Order

### Step 1. Reproduce on the current branch

Use the current app state and confirm the failure is truly case-independent:

1. load each mixed case
2. switch to `3D View`
3. capture whether the viewport stays at `Starting 3D renderer`, transitions to adapter error, or both
4. record browser capability state and exact error text

### Step 2. Confirm the failure is independent of sample-case data

Check whether the same 3D startup failure happens:

- on a local starter document with spaces
- across both `Active Floor Only` and `All Levels`

If the same error appears there too, keep `014.01`.

### Step 3. Separate `loading forever` from `late adapter failure`

If reproduction shows:

- the viewport stays in `Starting 3D renderer` and never resolves

then inspect the init promise path around:

- `loadRenderWasmModule()`
- `renderWasm.create_renderer(canvas)`
- cancellation and recovery effects

If reproduction shows:

- `Starting 3D renderer` first
- then the no-adapter error later

then treat that as one delayed renderer-init failure, not two different product bugs.

### Step 4. Fix the browser-side classification first when the underlying issue is no adapter

If the reproduction is a no-adapter environment problem:

- update `apps/web/src/three-d-viewport.tsx`
- map no-adapter messages to `unsupported`
- keep the existing `error` phase for actual unexpected renderer failures

### Step 5. Inspect the generated renderer artifact path before reverting feature work

If the reproduction is case-independent and began only after recent build or verify flows, inspect:

- `crates/render-wasm/pkg/*`
- `setup/ensure-render-wasm.mjs`
- `crates/render-wasm/src/lib.rs`

Restore or reuse the smallest proven known-good renderer path before considering broader reverts.

### Step 6. Only inspect sample-load plumbing if the failure is somehow tied to document replacement

If reproduction shows the failure appears only after sample-case loading and not on an equivalent local document, inspect:

- `apps/web/src/editor-shell.tsx`
- `apps/web/src/test-cases.ts`

Do not touch them otherwise.

### Step 7. Revert only the smallest proven cause

If a direct regression from `014.01` is proven, revert only the offending change.

If the real regression is in renderer startup or generated wasm artifacts, revert only that path instead.

Do not roll back the whole task unless the smallest selective fix still fails.
