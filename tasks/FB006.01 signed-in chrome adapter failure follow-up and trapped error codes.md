# FB006.01 Signed-In Chrome Adapter Failure Follow-Up And Trapped Error Codes

## Status

Implemented on `2026-03-27`.

Result:

- trapped web errors now surface stable `ER.md` codes in the UI so screenshots and copied messages can be mapped back to the registry quickly
- the `WEB-3D-002` overlay now exposes a `Retry 3D Startup` action so an already-open Chrome window can rerun renderer startup without a full page reload
- renderer startup now wraps browser `requestAdapter(...)` with a fallback chain so the signed-in Chrome window can recover by retrying softer WebGPU adapter strategies before surfacing `WEB-3D-002`
- the `WEB-3D-002` overlay now also exposes `Open 3D In New Window`, with a local handoff of the current editor state so a clean Chrome window can continue the same session
- startup no longer performs a separate adapter-null browser preflight, which cuts duplicated `No available adapters.` console noise in dev and keeps the real renderer path as the single adapter truth
- Windows adapter requests now strip `powerPreference` before calling `requestAdapter(...)`, so Chromium no longer prints the known ignored-option warning tracked in `crbug.com/369219127`
- Windows runtime attempts are deduped after option sanitization, so the browser no longer repeats `No available adapters.` for equivalent adapter requests that collapse to the same effective options
- the `WEB-3D-002` overlay detail is now short and user-facing: `No available adapter.` plus a direct fix path to `chrome://settings/system -> Use graphics acceleration when available -> Relaunch`
- the `WEB-3D-002` overlay now also exposes `Open Chrome System Settings` beside `Retry 3D Startup`, so the recommended browser recovery path is one click away from the failed viewport
- manual follow-up is still required in the exact failing Chrome window to confirm whether that browser context now starts 3D without the overlay

## Goal

Follow up on `FB006` now that new evidence shows the signed-in Chrome profile is still failing after the browser-side preflight was softened.

At the same time, make trapped errors easier to triage by showing the stable error code on product surfaces.

Keep this lean:

- no fallback renderer
- no scene-model change
- no auth or sample-data change unless evidence later forces it
- no Rust edit unless the renderer needs a more actionable machine-readable startup reason

## Current Evidence

The latest screenshot shows:

That wording comes from `create_renderer(...)` in `crates/render-wasm/src/lib.rs`, not only from the browser probe.

So the current signed-in Chrome failure is now confirmed as:

1. `navigator.gpu` exists
2. the app attempts real renderer startup
3. the renderer still cannot obtain a usable adapter in that Chrome context

### 2. The same machine still succeeds in another Chrome window/context

The comparison screenshot still shows:

- the same app URL
- the same machine
- a successful `3D View` in another Chrome window/context

That keeps the leading diagnosis browser-context-specific rather than project-data-specific.

Important nuance to preserve during follow-up:

- the original user report says an already-open Chrome window that is signed into Gmail fails
- a new Chrome window still works

So the first branch to verify is not only profile state. It is also whether this is an existing-window or browser-process-state issue inside the same profile.

### 3. The new detail text points at backend availability, not scene payload problems

The latest failing detail includes:

- `No suitable graphics adapter found`
- backend support lines such as `noop support not compiled in`, `vulkan support not compiled in`, `dx12 support not compiled in`, `gl support not compiled in`
- the earlier browser-probe diagnostic appended after the renderer failure

That means:

- the app is reaching the real wasm renderer startup path
- this is not a `WEB-3D-003` payload mismatch
- this is not a scene-serialization error
- the remaining issue is adapter availability inside the failing Chrome context

## Working Hypotheses

### 1. Signed-in Chrome profile state is blocking the same WebGPU path that succeeds elsewhere

The most likely current causes are:

- a stale or degraded GPU/browser-process state in the already-open Chrome window
- hardware acceleration disabled in that profile/window
- extension or policy interference in that profile/window
- browser GPU backend differences between the existing signed-in window and the fresh window

### 2. The app still needs a more deliberate unsupported-state recovery path

Right now the product can explain the failure better than before, but it still does not help the user retry or narrow the browser-context issue quickly.

Possible minimal follow-ups:

- add a small retry action on the `unsupported` 3D overlay
- let users rerun startup after changing browser settings without needing a full page refresh

### 3. Stable error-code surfacing should become the default for trapped messages

This diagnostic improvement is already worth keeping even before the root-cause renderer fix lands.

Reason:

- screenshots can now be matched to `ER.md` immediately
- copied messages remain human-first but no longer hide the stable registry code

## Investigation Plan

### Step 1. Reproduce with the new code-bearing message

Confirm the signed-in Chrome failure still lands on:

- `WEB-3D-002`

and capture whether the successful comparison window still reaches live `3D View`.

Do this with the smallest exact matrix first:

1. the already-open signed-in Chrome window that currently fails
2. a newly opened Chrome window using the same signed-in profile, if that is what the user is already seeing
3. only after that, any incognito, guest, or alternate-profile comparison

This ordering matters because a same-profile new-window success would point more strongly at window or browser-process state than at profile configuration.

### Step 2. Check the failing Chrome context outside the app

Use the failing signed-in Chrome context to inspect:

- hardware acceleration setting
- extension impact
- guest/incognito/profile difference
- `chrome://gpu` status

The goal is to prove whether the app is now only surfacing a real browser-context limitation or whether another app-side recovery path is still missing.

If the same profile succeeds in a newly opened window, prioritize:

- whether the failing existing window can recover after a local retry
- whether Chrome GPU-process restart behavior explains the difference better than profile settings do

### Step 3. Decide the smallest follow-up product fix

Choose only one of these paths:

1. If the issue is isolated to an already-open window or stale browser-process state:
   - add the smallest retry trigger on the `unsupported` overlay
   - keep `WEB-3D-002`
   - avoid over-blaming profile configuration in the recovery hint
2. If browser settings or extensions clearly explain the failure:
   - keep `WEB-3D-002`
   - tighten the recovery hint around that confirmed browser-context cause
3. If the renderer needs a more precise startup reason than the current free-form detail:
   - normalize the relevant Rust error text
   - keep the human summary and stable code in the web surface

## What Shipped

### 1. Registry-backed error codes now stay visible on web error surfaces

`apps/web/src/error-codes.ts`

- centralizes the `summary -> code` lookup for trapped web errors already registered in `ER.md`
- formats single-string messages as `Summary. Code: <CODE>.` before any `Detail:` suffix

`apps/web/src/App.tsx`

- login and auth surfaces now show code-bearing trapped errors

`apps/web/src/editor-shell.tsx`

- level-manager, logout, site setback, and site-plan banner errors now show code-bearing trapped errors

`apps/web/src/three-d-viewport.tsx`

- typed 3D overlays now show a dedicated `Code: <CODE>` line

### 2. `WEB-3D-002` now has an in-place retry path

`apps/web/src/three-d-viewport.tsx`

- when the viewport lands in `unsupported` with `Code: WEB-3D-002`, the overlay now shows `Retry 3D Startup`
- the action frees any stale renderer handle, clears the issue, and reruns startup without requiring a page refresh

This is intentionally narrow:

- no fallback renderer
- no Rust change
- no retry button for the missing-API `WEB-3D-001` case

### 3. wasm startup now retries softer adapter strategies before failing

`apps/web/src/three-d-viewport.tsx`

- browser-side adapter startup now builds a fallback chain from the renderer's original request through softer options before letting wasm surface `WEB-3D-002`
- the wrapped startup path now tries the original request first, then browser-default and fallback-adapter variants when needed
- on Windows, the effective runtime request strips `powerPreference` before calling the browser API, matching Chromium's current behavior and avoiding the ignored-option warning
- duplicate runtime attempts are skipped after that Windows sanitization so equivalent adapter requests do not spam browser warnings
- renderer startup still keeps internal diagnostics for classification, but the `WEB-3D-002` overlay detail is now reduced to a short user-facing recovery message instead of raw adapter/backend text
- the typed overlay now keeps the recovery actions together in one row: retry startup, open `chrome://settings/system`, or hand off the same editor state into a clean window
- a small deferred startup guard avoids the duplicate strict-mode mount from firing duplicate adapter requests during dev startup

`apps/web/src/three-d-viewport.test.ts`

- adds pure tests for the fallback-chain builder, Windows option sanitization, deduped runtime attempts on Windows, and the recovery path where a later adapter attempt succeeds after the first strict request returns `null`

### 4. `WEB-3D-002` can hand off the live editor state to a clean window

`apps/web/src/editor-shell.tsx`

- stores a short-lived handoff payload in `localStorage` when the user requests a fresh 3D window from the unsupported overlay
- restores `ProjectDoc`, active level, view, selection, sample-case id, and 3D scope from the handoff token in the new window
- removes the handoff token from both `localStorage` and the URL after restore so stale links do not linger
