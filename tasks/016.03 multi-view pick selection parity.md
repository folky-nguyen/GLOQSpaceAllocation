# 016.03 Multi-View Pick Selection Parity

## Goal

In `apps/web`, keep the current click-based multi-pick workflow for the current selectable `space` element and extend it beyond `Floor Plan` so `Site Plan` and `3D View` can also pick many visible spaces through the generic selection seam introduced in `016.02`.

## Required Outcome

- keep click-based multi-selection working on `Floor Plan`
- allow click-based multi-selection for visible site-plan spaces
- allow click-based multi-selection for visible spaces in `3D View`
- keep the current selection emphasis and properties behavior working after the parity update
- keep current supported selectable element kinds limited to `space`
- leave a clean extension path so future 3D-selectable element kinds can plug into the same seam later without reopening the selection shape again

## Constraints

- keep the change local to `apps/web`
- TypeScript must continue owning authored pick and selection behavior
- do not move selection rules or authored hit logic into the wasm renderer
- do not change `ProjectDoc`, persistence, Supabase schema, or API contracts for this task
- keep `site-edge` setback editing working in `Site Plan`
- do not reintroduce sweep-select or a second selection mode

## Current Repo State

### `apps/web/src/ui-store.ts`

Today the store already has the generic selection seam from `016.02`:

- one selected `element`
- many selected `element-set`
- current supported `SelectionElementRef` kind: `space`

That seam is ready for multi-view reuse, but only `Floor Plan` currently drives it through click-pick.

### `apps/web/src/editor-shell.tsx`

Today `Floor Plan` already supports click multi-pick through `toggleSpaceSelection(...)`.

`Site Plan` does not yet match that behavior:

- site spaces render as `.site-plan-space`
- the SVG group is not clickable
- `.site-plan-space` is explicitly blocked with `pointer-events: none`
- site edges still have their own click target for setback editing
- `Select All Visible` still hard-switches to `Floor Plan`, so click parity is the leaner target than menu parity for this task

### `apps/web/src/three-d-viewport.tsx`

Today `3D View` shows selection emphasis, but it does not have a click-pick path yet:

- pointer input is used for orbit, pan, and zoom
- there is no click-to-select callback back into the shell
- there is no TypeScript-side hit test for visible space prisms

### `apps/web/src/space-scene.ts`

Today 3D already derives:

- visible space prisms
- current camera frame and view-projection helpers
- selected-space emphasis

That makes it a good place to host small shared helpers for `3D View` picking without pushing authored interaction into wasm.

## Scope

In scope:

- enabling site-plan space click selection through the existing generic element selection seam
- adding a `3D View` click-pick path that can toggle visible space selection
- keeping multi-pick behavior consistent with `Floor Plan`
- preserving `site-edge` selection in `Site Plan`
- keeping the current supported selectable element kind limited to `space`

Out of scope:

- adding new non-space selectable element kinds
- reworking the renderer contract just to support picking
- adding drag-select, lasso-select, or area-select behavior
- changing persistence, API, or Supabase storage
- redesigning the `Select` menu beyond what the parity change strictly needs
- perfect occlusion-accurate 3D picking or renderer readback

## Initial Implementation Direction

### 1. Add `Site Plan` click parity for spaces

Lean rule:

- make site-plan spaces clickable
- route site-plan space clicks through the existing `toggleSpaceSelection(...)` flow
- keep `site-edge` click behavior intact so setback editing still works

Reason:

- this is the smallest path to match `Floor Plan` behavior without inventing a second site-plan selection flow

### 2. Add one shell-owned `3D View` pick callback

Lean rule:

- let `three-d-viewport.tsx` report a picked visible element back to `editor-shell.tsx`
- keep the authored selection toggle in the shell, not in wasm

Reason:

- the shell already owns selection state and multi-pick behavior
- the viewport should only report what was hit, not decide how selection mutates

### 3. Extract one shared visible-space helper before adding 3D pick math

Lean rule:

- pull the visible-space filtering and per-space height lookup into one small helper in `space-scene.ts`
- let both scene building and later 3D picking read from that same visible-space seam

Reason:

- this is smaller than duplicating visibility and level-height logic inside `three-d-viewport.tsx`
- it keeps 3D pick behavior aligned with what the renderer is already showing

### 4. Add minimal TypeScript hit-testing for visible spaces

Lean rule:

- keep the 3D pick helper in TypeScript
- use the shared visible-space seam plus current camera state to pick one clicked visible space
- prefer one first-pass screen-space or projected-face hit test over a broader general-purpose prism picking system

Reason:

- current space geometry and camera helpers already live in TypeScript
- this keeps the renderer contract lean while leaving room for future 3D-selectable element kinds

### 5. Treat click and camera drag as different intents

Lean rule:

- only treat a pointer interaction as selection when it stays within a small movement threshold
- keep orbit, pan, and zoom behavior unchanged for real camera gestures

Reason:

- `3D View` already uses pointer input for navigation, so pick parity needs an explicit click-vs-drag guard

## Review Update

The first draft was still broader than necessary for the current repo size because it implied a more general `3D` prism hit-testing pipeline and left room to duplicate visibility logic outside `space-scene.ts`.

After reviewing the current seams in `editor-shell.tsx`, `space-scene.ts`, and `three-d-viewport.tsx`, the leaner implementation path is:

- keep the user-facing goal narrowed to parity for current selectable `space`s only
- keep `Select All Visible` out of scope for `016.03`, because the user asked for pick-many parity and the menu still hard-switches to `Floor Plan`
- reuse the existing generic `SelectionElementRef` seam from `016.02` without changing its shape again
- extract one small visible-space descriptor helper in `space-scene.ts` so rendering and 3D picking share the same visible-space source of truth
- add a first-pass `3D View` pick helper in TypeScript only, using the shared visible-space seam and current camera state
- explicitly accept a lean first pass instead of perfect hidden-surface picking, as long as common visible-space clicks work and camera drag remains intact

This keeps the implementation local to the web shell and should minimize code churn across `editor-shell.tsx`, `space-scene.ts`, and `three-d-viewport.tsx`.

## Acceptance

- `Floor Plan` multi-pick still works
- clicking site-plan spaces can add and remove them from the current selection

- `3D View` camera drag behavior still works after adding picking
- no new renderer API or wasm-owned selection behavior is required for the first pass
- the implementation still supports only `space` as the current selectable element kind
