# 006.01 Email Password Auth In Web

## Goal

Update the existing `apps/web` Supabase auth flow so the default sign-in path uses email + password.

Required behavior:

- app load still restores an existing session
- `/login` signs in with email + password
- `/editor` stays protected
- logout still works
- OTP is no longer used for normal login
- OTP is used only for:
  - new account confirmation
  - forgot-password recovery

Optimization goal:

- make the smallest code change that still covers the requested behavior
- reuse the current auth module, current routes, and current auth card instead of introducing a larger auth subsystem

Non-goals:

- no Rust API auth changes
- no social auth
- no profile/settings work
- no extra frontend state library
- no new folder structure for auth

## Current Repo State

The repo already completed the original auth split from task `005`.

Relevant files today:

- `apps/web/src/auth.ts`
  - already owns the Supabase client, auth snapshot, bootstrap, and logout
  - already has `pendingEmail`
  - is the right seam to keep using
- `apps/web/src/App.tsx`
  - already owns `/`, `/login`, and `/editor`
  - already renders one public auth screen
- `apps/web/src/editor-shell.tsx`
  - already renders the signed-in shell and logout affordance
- `apps/web/src/ui-store.ts`
  - still owns editor chrome state only
  - still lacks one explicit reset helper for logout cleanup
- `apps/api/src/auth.rs`
  - already verifies Supabase Bearer access tokens

This means the API does not need to change. Only the browser-side login UX changes.

## Minimal-Change Strategy

### 1. Keep the current route shape

Do not add `/signup` or `/forgot-password`.

Keep only:

- `/`
- `/login`
- `/editor`

Handle the public auth flows inside `/login` with a local mode switch:

- `"login"`
- `"signup"`
- `"recovery"`

Reason:

- this is the smallest diff from the current app
- it avoids adding more route components, redirects, and page-level styling branches

### 2. Keep one auth module and change only the auth actions

Keep `apps/web/src/auth.ts` as the only auth module.

Preserve:

- Supabase client creation
- env validation
- `bootstrapAuth()`
- `useAuth()`
- browser-side auth bypass
- logout

Replace only the passwordless helpers with password-based helpers.

Smallest useful public surface:

```ts
export type AuthStatus = "loading" | "signed_out" | "signed_in";

export type AuthSnapshot = {
  status: AuthStatus;
  session: Session | null;
  user: User | null;
  error: string | null;
  pendingEmail: string;
  pendingOtpType: "email" | "recovery" | null;
  recoveryReady: boolean;
};

export function bootstrapAuth(): Promise<void>;
export function useAuth(): AuthSnapshot;
export async function signInWithPassword(email: string, password: string): Promise<{ error: string | null }>;
export async function signUpWithPassword(email: string, password: string): Promise<{ error: string | null }>;
export async function sendRecoveryEmail(email: string): Promise<{ error: string | null }>;
export async function verifyEmailOtp(
  email: string,
  token: string,
  type: "email" | "recovery"
): Promise<{ error: string | null }>;
export async function updatePassword(password: string): Promise<{ error: string | null }>;
export async function logout(): Promise<{ error: string | null }>;
```

Reason:

- this stays very close to the current shape
- `pendingEmail` is already present, so the OTP flows can reuse it
- one extra `pendingOtpType` field is enough to distinguish signup vs recovery

### 3. Keep `/login` as one card with conditional sections

Refactor the current `LoginPage` instead of splitting it into multiple public pages.

Recommended UI:

- one shared auth card
- one small mode switch row
- one shared error region
- one local `action` state for loading labels

Mode behaviors:

- `login`
  - email field
  - password field
  - `Log in`
  - secondary buttons: `Create account`, `Forgot password`
- `signup`
  - email field
  - password field
  - `Create account`
  - if confirmation is required, show OTP field and `Verify code`
- `recovery`
  - email field
  - `Send recovery code`
  - then OTP field and `Verify code`
  - after OTP success, show new password field and `Set new password`

Reason:

- this keeps almost all auth UI work inside one existing component
- it minimizes CSS churn
- it avoids duplicating shared card/layout/error markup

### 4. Primary login switches to password auth only

Use:

```ts
supabase.auth.signInWithPassword({ email, password })
```

Remove from normal login:

- `Send link or code`
- `Verify code`
- passwordless sign-in copy

Reason:

- this is the actual user requirement
- it is the smallest behavioral change on the main path

### 5. Signup stays simple: password first, OTP only if confirmation is required

Use:

```ts
supabase.auth.signUp({ email, password })
```

Expected behavior:

1. user enters email + password
2. app calls `signUp`
3. if Supabase returns a session, auth completes immediately
4. if Supabase requires email confirmation, store:
   - `pendingEmail`
   - `pendingOtpType: "email"`
5. show OTP input in the same `/login` card under signup mode
6. verify with:
   - `supabase.auth.verifyOtp({ email, token, type: "email" })`

Reason:

- no extra callback screen is needed
- signup still supports OTP-only confirmation when the project is configured that way

### 6. Forgot password stays in the same card

Start recovery with:

```ts
supabase.auth.resetPasswordForEmail(email, { redirectTo })
```

Then keep the UI in recovery mode:

1. user enters email
2. app sends the recovery email/code
3. store:
   - `pendingEmail`
   - `pendingOtpType: "recovery"`
4. user enters OTP in the same card
5. verify with:
   - `supabase.auth.verifyOtp({ email, token, type: "recovery" })`
6. once verified, reveal a new password field
7. finish with:
   - `supabase.auth.updateUser({ password })`

Important note:

- this recovery flow depends on Supabase email templates and redirect setup supporting a code/token-based recovery experience

Reason:

- it satisfies the requested “OTP only for forgot password”
- it keeps the code path inside existing auth seams

### 7. Protected route behavior stays unchanged

`/editor` should keep the current three-state behavior:

1. `loading`
2. `signed_out`
3. `signed_in`

Do not move auth checks into `editor-shell.tsx`.

Reason:

- this already works
- no reason to enlarge the diff

### 8. Add one tiny logout reset hook for editor chrome state

Add a small reset helper to `apps/web/src/ui-store.ts` and call it after successful logout.

Reset only:

- active view
- select mode
- selection

Reason:

- this is a very small change
- it prevents auth-session UI leakage without adding a larger architecture change

## Experience Notes

- Keep auth state out of Zustand.
- Keep browser env names unchanged.
- `VITE_LOCAL_AUTH_BYPASS` may be enabled temporarily in local or deployed builds when the client should enter `/editor` directly without showing the sign-in screen.
- If `VITE_SUPABASE_URL` and `VITE_SUPABASE_PUBLISHABLE_KEY` are intentionally left unset for a temporary no-auth deployment, the app should also fall back to direct workspace access instead of rendering the sign-in card with a missing-env error.
- OTP verification must still work if the user refreshes the page and re-enters the email manually.
- Login errors should stay generic.
- `signUp()` returning no session is a valid “confirm with OTP next” outcome, not an error.
- Recovery should not show the new password field until OTP verification succeeds.

## Supabase Project Setup Notes

This plan stays small in code, but it assumes the Supabase project is configured to support the requested UX.

### Email auth settings

- Email + password sign-in must be enabled.
- If email confirmation is enabled, signup may require OTP confirmation after `signUp()`.

### Templates and redirects

To keep OTP only for signup and forgot password:

- the signup email template must expose a usable token/code flow
- the reset-password email template must also support a token/code flow
- local redirect URLs should continue to include the existing app origins for `5173` and `3001`

If the project keeps a pure magic-link recovery template, the code can still stay small, but the UX will not match this task exactly.
